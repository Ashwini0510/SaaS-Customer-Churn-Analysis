# Case Study: Customer Churn & Revenue Rescue Analysis (SaaS)

## 📌 Business Overview & Context
In a subscription-based Software-as-a-Service (SaaS) business model, sustaining a healthy growth rate relies heavily on maximizing customer lifetime value and maintaining low customer churn. This case study analyzes internal data from a high-growth Project Management SaaS application that recently experienced an unexpected spike in monthly contract cancellations.

As the **Business Analyst**, my objective was to perform a data audit on our customer database to quantify the exact financial impact of this churn, isolate risk trends across customer subscription tiers, and construct a proactive, data-driven "Risk List" to empower our Customer Success team to save accounts before they cancel.

## 🗃️ Database Architecture & Schemas
The analysis evaluates operational backend data across two key database tables:

### 1. `subscriptions` (Customer Account Ledgers)
| Column Name | Data Type | Description |
| :--- | :--- | :--- |
| `customer_id` | INT (PK) | Unique identifier for the corporate client account |
| `customer_name` | VARCHAR | Name of the enterprise/organization |
| `tier` | VARCHAR | Contract tier level ('Basic', 'Premium', 'Enterprise') |
| `monthly_amount` | INT | Monthly Recurring Revenue (MRR) generated by the account |
| `signup_date` | DATE | Date the customer initiated their active contract |
| `status` | VARCHAR | Operational status ('Active' or 'Churned') |
| `churn_date` | DATE | Date of cancellation (Null if account is still active) |

### 2. `usage_logs` (Product Telemetry Logs)
| Column Name | Data Type | Description |
| :--- | :--- | :--- |
| `log_id` | INT (PK) | Unique identifier for the logging entry |
| `customer_id` | INT (FK) | Maps activity logs directly back to the subscription account |
| `last_login_date` | DATE | Most recent date a user from the organization accessed the app |
| `features_used` | INT | Cumulative count of unique functional modules deployed in the last 30 days |

## 🕵️‍♂️ Key Analytics Challenges & SQL Approaches

### Challenge 1: Quantifying the Financial Damage (Lost MRR)
* **Business Objective:** Executive leadership required an exact monetary figure representing the scale of monthly contract cancellations to recalibrate financial forecasting.
* **SQL Strategy:** Calculated the aggregate monthly financial leak by running a `SUM` function across the revenue column, strictly filtering for inactive account statuses.

```sql
SELECT SUM(monthly_amount) AS total_lost_mrr
FROM subscriptions 
WHERE status = 'Churned';
